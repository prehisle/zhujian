# 设计铁律(勿违)— 朱笺(zhujian)

> 当前世界观(进度㉖ 定调,㉜ 收成单实体):**纯人工 + 灵活不受限 + 维护性高,做减法**——存储层只保「数据不丢失」类保证(㉜ 起为 0014 的 `item_revisions` 历史 + `trg_item_no_delete_live_organized` 删除守护),砍掉所有「流转锁死」的触发器(几乎都是为防 AI/系统静默改数据,AI 撤走后只锁住用户自己又增维护成本)。

- **单实体:想法与待办是同一行 `items` 的不同 `stage`**(进度㉜,codex 评 0014 数据模型 GO):`notes`+`tasks` 已物理合并为一张 `items`,`stage ∈ (inbox 未归类灵感 / filed 已归类灵感 | todo / doing / confirming / done)`。**转待办=翻 stage 到 todo(零副本)**,撤回=翻回 inbox/filed;**任务态条目不出现在灵感视图**(天然去重——旧双实体「转待办留两条同内容记录」的违例就此消失)。`archived_at` 是回收站轴(冻结 stage、restore 回原 stage);标签 `item_topic` M:N(想法+任务统一,**㉞ board 已是真多标签**:`add_task_topic`/`remove_task_topic`,一个任务可多个标签);历史 `item_revisions` 覆盖全 stage(改任务标题也归档)。铁律仍是「最小化/同一主体」——一个主体一行,跨灵感与任务的统一生命周期。
- 风格:轻量 / 最小化 / **绝不回退兜底**(fail-fast,产品代码不写静默默认值)。
- **(㉜ 起)`items` 是唯一事实源**(原 `notes`/`tasks` 已合并);`topics` 是派生标签视图,**进度㉖ AI 删除后纯由用户手动维护,无「自动重算覆盖用户数据」风险**。topics 的「可重算」由手动合并(`notes::merge_topics`,见进度⑯)行使:重写 `item_topic` 投影(㉜ 起单次 repoint 同时覆盖想法+任务的标签)、删空源主题、不保留 membership 历史,不丢 item/不碰 content。**task 自此无 `suggested` 态、无 AI 重算**——原先为防 AI/系统静默改 task 而设的 0002/0005/0006/0009 存储层触发器(suggested 闸门、归档状态/元数据冻结、删除守护)已被 **0010 重建表时全部移除**;task 的「不变量」(分级删除、撤回为灵感[仅待办]、归档 restore 回原列)改由**命令层事务**守护(只保数据不丢失,不再用触发器锁流转,见进度㉖)。
- **任务看板四态、可选非强制**(进度㉙):状态机是 {todo, doing, confirming, done} **全连通**(任意态互相流转、self-move 非法),`task::STATES` 常量是单一真相源(`legal` + 拖动列守卫共用,加态只改一处)。**「待确认」(confirming)是可选去处、不是关卡**——`进行中→已完成` 快捷路径保留,待确认只承接「做完但在等外部确认/回音、需从进行中分离开」的异步工作,契合「灵活不受限、不强制流转」。新增 `migrations/0013` 沿用 0010「重建 tasks 表」手法改 status CHECK(纯加成员,无状态需转换)。
- **`task.topic_id` 是「任务看板当前主题指针(用户标签)」**(进度㉑):与 note 的 topics(`note_topic` 投影)隔离——不反推 note。删主题 FK `ON DELETE SET NULL` 把指向它的(活跃+归档)task 落回无主题;并主题 `repoint_task_topic` 重指(含归档任务);用户**手动**改标签经 `set_task_topic` 的 `archived_at IS NULL` 命令层守卫禁于归档态。**历史注(进度㉑→㉖)**:0007 当初有意不给 topic_id 加归档冻结触发器(SQLite 的 FK 动作本身即触发器程序,加冻结会让删/并主题在归档任务上 RAISE,是 codex 一轮 NO-GO 的根因);该取舍随 0010 清掉全部 task 触发器后已无关——现在 task 上没有任何存储层触发器,归档期约束(如冻结)仅靠命令层守卫。
- **不可变性是历史级,不是行级**(2026-06-15 修订,codex GO;㉜ 升级为 item 版、覆盖全 stage):用户**可以编辑**自己的条目,每次改动前的旧版本由 DB 触发器 `trg_item_archive_on_edit`(㉜ 0014,原 0003 note 版升格)自动归档进 `item_revisions`(append-only;`trg_item_revision_immutable` 禁改写)——**任何代码路径都不可能覆盖 content 而不留历史**,原文与每一版永不丢失。**㉜ 起覆盖全 stage**:改任务标题(`rename_task`)也归档历史,不再是想法专属。**编辑是 user-only**(进度㉖ AI 已物理删除);archived(回收站)的条目冻结不可编辑。标签表 `item_topic` 只挂 item_id 不挂版本——编辑后是**语义可能漂移、非引用脱节**,这是单用户自负的有意取舍。
- **删除是用户主权、且分级 fail-safe**(2026-06-15 修订,codex 两轮 GO,见进度⑫;**㉜ 收成单实体后机制统一**):铁律本意是"防**系统/AI 静默**毁掉/覆盖用户数据、防意外丢失",**不是禁止用户本人显式清理自己的数据**。**㉜ 起单一机制**:删除守护 `trg_item_no_delete_live_organized`(0014,`BEFORE DELETE ON items`)——**仅 `stage='inbox' 且未归档`可硬删**(捕获箱丢垃圾,无派生结构);`filed`/任务态 live 直删 ABORT,必须先软删进回收站(盖 `archived_at`)、回收站再二次确认 `DELETE`(cascade `item_topic`/`item_revisions`)。灵感与任务**共用这一条 `archived_at` 轴**,前端按冻结 stage 分「灵感回收站/任务回收站」两 tab。**73(2026-07-08)UI 收敛:删除=进回收站,销毁只在回收站里发生**——灵感「删除」不再按 stage 分流,inbox 也软删进回收站(`repo::archive_idea` 放宽收 inbox+filed;可还原故零确认,还原回冻结的原 stage)。用户以真实体感拍板(「只要是删除就该能反悔」),且同步语义放大了硬删代价:tombstone=全网抹掉、不可复活,不该是删除键的默认归宿——**反转的是 65 拍板里「UI 保留 inbox 直接硬删」那半句;存储层删除守护与墓碑契约原样不动**(inbox 硬删仍是合法原语,`delete_note` 留给命令层/e2e 清库;回收站「彻底删除/清空」仍逐条发 tombstone、delete-wins-sticky 照旧)。**㉜ 前的旧表述(下文 ① processed/0004/task_note/1:1 task 存活)为双实体历史**——单实体下 purge 一条就是删那一行,无「1:1 task 残留」之说。① inbox junk 直接硬删(无派生结构);② ~~**processed 不可*直接*硬删**~~(是溯源根)——`0004` 触发器在存储层 `BEFORE DELETE ON notes WHEN OLD.status='processed' RAISE(ABORT)` 兜住任何路径;要删必须**两段式**:已整理【删除】=软删 `processed→archived`(进回收站、可还原、provenance/历史全留),回收站【彻底删除】=显式二次确认才真 `DELETE`(cascade note_topic/task_note/note_revisions)。关键:`notes` 是 FK 根、**无任何路径会 cascade-delete 一个 notes 行**,故 0004 只拦直删、不误伤合法级联。彻底删除会销毁该 note 的编辑历史——这是用户**显式**销毁自己数据的合理边界。**purge 后**:note 被彻底删,cascade 掉它的 note_topic / task_note link;**进度㉖ 起想法↔待办 1:1、无 AI `suggested` 任务,故无 orphan 清扫**;与它 1:1 的用户态 task 仍存活(只断 provenance link——要把任务连根撤掉走「撤回为灵感」`revert_task_to_inbox`,仅限待办)。③ **task 也分级**(进度⑬→㉕→㉖):任意活跃任务(todo/doing/done)【删除】=软归档,盖 `archived_at` 进回收站(可还原,删个 todo 还原仍回 todo 原列),回收站【彻底删除】=显式二次确认才真 `DELETE`。**进度㉖ 注**:这套分级原先由存储层触发器兜底(0002/0005/0009),但那些触发器已随 0010 重建表移除——现在**分级删除、归档期冻结、restore 回原列全由命令层(`task::archive`/`restore`/`purge`)事务守护**(`restore` 读出冻结 status 再重设该列末尾 position);`suggested`/`dismiss` 概念已不存在。**注**:`task.position` 从不是硬 dense 不变量(0008 起 partial unique 只保证 active 列内唯一;**66/0021 起 position 是 fractional index 文本键**[frindex.rs base62,字节序即列序],拖动只改被拖卡的键),归档/跨列腾空留洞无害。**④ 成就归档不可删(55,0017 `sealed_at`)**:与回收站平行且互斥的第二根轴——回收站=「我不要了」,归档=「我干完了、留着看」的**史实**(「历史级不可变」哲学的自然延伸)。只有活跃 done 可归档;归档后整行冻结(`trg_item_sealed_frozen`)、硬删软删一律 ABORT(`trg_item_sealed_no_delete`)、归档视图**不给删除入口**。删除主权不废除、只加一步:取消归档(翻回 done 列尾)→ 再走正常两段式删除——冲动删不动,主权仍在。内部命名 seal(封存)刻意避开被回收站占用的 archive_*,「归档」一词在用户可见层只有成就册一个意思(概念隔离)。
- **灵感与待办是同一主体的两档成熟度**(进度㉘ 定调):灵感 = 还没想清楚的任务,待办 = 想清楚了的灵感。故「撤回为灵感」不按「出生地」(是否从想法转来)区别对待,而按**当前状态**——只有最不成熟的 **待办** 档可退回更早的灵感;**进行中/已完成** 已是想清楚的任务,退回自相矛盾,不给此能力。**㉜ 单实体后落地大幅简化为「同一行翻 stage」**(`notes::revert_task_to_idea`/`repo::revert_to_idea`):todo→inbox/filed,**还挂标签则落已归类(filed)、否则回未归类(inbox)**,同时清掉任务专属属性(position/due/priority,灵感态列内 CHECK 要求 NULL)。**不再有「还原源想法 vs 按标题播种新想法」分叉**——本就是同一行,没有第二条记录。撤回是命令层事务、fail-fast 回滚。
- **纯人工**(进度㉖ 定调):想法的完整流转(编辑/转待办/归主题/合并/撤回/删除)全靠手动主线(`notes.rs`/`task.rs`),不依赖任何 AI——AI 整理/补提代码已物理删除。要再用 AI(决策辅助/总结)需从头加。
- 概念隔离:`items.stage`=流程档位(灵感↔任务)/ `topics`+`item_topic`=标签(进度㉚ 对用户从「主题」重定位为「标签」——轻量分类 + 下钻聚合,底层表名/命令/`topic_id` 不变,同一概念换更准的中文名不违隔离;**㉞ `summary` 字段/列已彻底删除(0015)**;`rejected_suggestions`「agent 记忆」表已随 AI 删除)。
