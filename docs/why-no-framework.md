# 决策:前端不用框架(vanilla TS + Vite)

> 2026-06-29 定。背景:用户问「没有框架、是不是 Vue/React 就能自动局部刷新、避免那次 alt-tab 闪烁?」并交代「保持高可维护性,日后若切框架也相对轻松;先自用,有价值再开源、产品化」。

## 决定

前端保持 **vanilla TypeScript + Vite,不引入框架**(Vue/React/Svelte 等)。

## 为什么

- **项目定位**:单人、本地、轻量的 Tauri 工具。世界观是「纯人工 + 灵活不受限 + 维护性高,做减法」(见 [design-rules.md](design-rules.md))。引框架是一次全量重写 + 长期扛构建链/依赖/版本/打包体积,**收益配不上代价**。
- **那次闪烁不是「没框架」的必然**:根因是渲染策略图省事——每次刷新 `replaceChildren` 整块重建。框架的虚拟 DOM/响应式会自动「数据没变就不动节点」,而我们用十几行「**算数据指纹,没变就跳过整块重绘**」补上了同一类保护(粗粒度版;`board.ts::load` / `inbox.ts::refresh` 的 `refocus` 跳过逻辑)。框架做得更细,但这个体量不需要。
- **差异化**:满市场都是「AI 帮你管一切」,一个明确「不靠 AI、轻、不打扰」的工具反而稀缺——这是产品身份,值得守。

## 怎样让「日后切框架」保持便宜(免费纪律,不是额外脚手架)

让迁移痛不痛的,**不是「预留抽象层」(那本身就是臃肿),而是保持干净分层**:

- **能平滑迁移 / 真正的资产**:Rust 后端 + SQLite + 命令契约(`list_tasks`/`capture_note`…)与前端框架无关,换框架一行不改;以及「数据 → 该显示什么」的**纯计算**(如「任务 + 筛选 → 要展示的卡片」,现已是纯函数)。
- **注定丢弃 / 一次性消耗品**:手写 `replaceChildren` 拼 DOM、事件绑定、`hotkey-menu.ts` 直接操作 DOM 的部分——**这正是框架要替你做的事**,别费劲把它做「可移植」,否则等于自造小框架。
- **唯一要守的纪律**:别让业务规则渗进拼 DOM 的代码里;数据怎么取、规则怎么算,与「长成什么 DOM」分开放。做到这点,换框架就是局部、可控的改动——而这本就是好代码的样子,非额外负担。

## 何时重新考虑

- 前端长出**大量精细、状态密集的交互**,手写 DOM 更新明显拖慢迭代时——届时可先评估「几十行小 diff 工具」这种更轻的中间档,再谈整套框架。
- 真要从「本地单人工具」走向「产品」时:那一步(多设备同步/账号/云端)比换框架大得多,会先逼数据层重构。**现在一概不建**——等真有跨设备需求再说,别为不存在的需求买单。
